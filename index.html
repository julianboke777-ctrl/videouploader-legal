Video Uploader

A desktop automation tool that publishes self-produced video content to TikTok creator channels owned and operated by the developer, using the official TikTok Content Posting API.

Version 1.0 · Updated 2026-05-17

What it is

Video Uploader is a small, locally-run desktop application built and operated by a single developer. Its sole purpose is to publish pre-rendered MP4 video files to TikTok creator accounts that the developer owns and operates — replacing the manual TikTok Studio web upload with a scripted, reproducible workflow.

There are no end users other than the developer. The tool is not distributed publicly, is not offered as a service, and does not handle any third-party content or accounts.

In one sentence

Local script → OAuth Login Kit → Content Posting API (Direct Post) → chunked FILE_UPLOAD → status poll until PUBLISH_COMPLETE.

Who it’s for

One audience: the operating developer, publishing to their own creator channel(s). The tool is not multi-tenant, has no user accounts, no UI for external users, and does not expose any network service.

Key features

Direct Post via official API

Uses TikTok’s Content Posting API (Direct Post) endpoint /v2/post/publish/video/init/. No third-party intermediaries (no Buffer, Hootsuite, Zapier, etc.).

Chunked FILE_UPLOAD

Videos are uploaded as chunks via PUT from the developer’s local machine. PULL_FROM_URL is intentionally not used.

OAuth via Login Kit

One-time OAuth flow authorizes the developer’s own creator account. Scopes: user.info.basic (open_id mapping) and video.publish.

Local-only token storage

Access + refresh tokens are stored in a single file on the developer’s machine with file permission 0600. Refresh tokens are rotated on every refresh, per TikTok’s specification.

Low cadence by design

Typical posting rate is 1–2 videos per day, per channel. No bulk posting, no batched flooding, no scheduled spam.

No analytics scraping

The tool does not read user data, follower lists, comments, or analytics. The only API surface used is video publishing + the minimum OAuth identity scope.

How it works

1
Local rendering. The developer produces a 60-second portrait MP4 locally (H.264, 1080×1920, 25 fps). The tool does not generate or modify video content.
2
One-time OAuth. The developer runs oauth_bootstrap.py, which opens the TikTok authorization page in the default browser. After clicking Allow, an access token and refresh token are written to a 0600-permissioned file at ~/.claude/.config/tiktok-tokens.json.
3
Init upload. The tool calls /v2/post/publish/video/init/ with source=FILE_UPLOAD, providing the file size, chunk size, and total chunk count. TikTok returns a publish_id and an upload_url.
4
Chunked PUT. The tool uploads each chunk to upload_url via standard HTTP PUT with the correct Content-Range headers. Files under 5 MB use a single chunk.
5
Status poll. The tool polls /v2/post/publish/status/fetch/ until status becomes PUBLISH_COMPLETE (or fails with a documented error code, which the tool surfaces unmodified).
6
Refresh rotation. When the access token expires (24 h TTL), the tool exchanges the refresh token for a new access token and a new refresh token, persisting both back to the 0600 file. No re-authorization is required.

Privacy & security posture

  • No data collection. The tool does not log, store, or transmit user data beyond what is strictly required to publish a video (the video file itself, a caption, and the requested privacy_level).
  • No third-party services. The integration is direct: the developer’s machine talks to open.tiktokapis.com over HTTPS. No middleware, no analytics SDKs, no telemetry.
  • Local-only credentials. Client Key, Client Secret, access tokens, and refresh tokens never leave the developer’s machine. All credential files are mode 0600 (owner read/write only).
  • Refresh rotation enforced. Every successful refresh returns a new refresh token, which is persisted immediately. Failure to rotate would lock the tool out, by design.
  • Token revocation. The developer can revoke access at any time from TikTok’s account settings → Privacy → Manage apps and websites → Video Uploader → Remove access.

Use-case summary for review

Single-developer, single-purpose automation for owner-operated creator channels. The application uploads self-produced 60-second portrait videos to creator accounts owned by the developer at a cadence of 1–2 videos per day per channel.

Each upload is initiated explicitly by the developer through the local script. The application calls /v2/post/publish/video/init/ with FILE_UPLOAD source, uploads the video chunks via PUT to the returned upload_url, then polls /v2/post/publish/status/fetch/ until PUBLISH_COMPLETE.

OAuth tokens (access + refresh) are stored locally with file permission 0600. Refresh tokens are rotated on every refresh as required by the spec. The application does not store user data, does not collect analytics, and does not interact with any TikTok user other than the authenticated creator account.

The application does not use third-party services (Buffer, Hootsuite, etc.) — it is a direct integration via the official Content Posting API.

FAQ

Is this app available to download?
No. It is a private, single-developer tool. There is no public release, no installer, and no website signup.
Does it post to other people’s accounts?
No. It posts only to creator accounts owned and operated by the developer, authorized via a one-time explicit OAuth flow.
Does it use PULL_FROM_URL?
No. Only FILE_UPLOAD is used, with the video uploaded as chunks via PUT from the developer’s machine. PULL_FROM_URL would require domain verification of an external host, which is intentionally avoided.
What happens if the API returns an error?
The script surfaces TikTok’s documented error response unmodified and exits with a non-zero status. No retry storm, no silent swallow.
How are tokens stored?
In a single JSON file on the developer’s local disk, with POSIX mode 0600 (owner read/write only). Never sent anywhere except back to TikTok’s token endpoint during refresh.
Why Desktop platform and not Web?
The tool is a local script. There is no hosted web frontend, no inbound network service, and no UI exposed to remote users. Desktop is the accurate platform classification.
How is the integration audited?
The endpoints used, scopes requested, and file-handling behavior are documented in full on this page and in the privacy policy. The integration is direct (no third-party SaaS in the path).
